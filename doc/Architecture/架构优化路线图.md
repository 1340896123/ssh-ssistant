# SSH Assistant 架构优化路线图

> 版本：v1.0 | 日期：2026-04-18 | 状态：规划中

## 1. 现状量化评估

### 1.1 前端度量

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| 最大组件行数 | 2348 (FileManager.vue) | < 500 |
| invoke 命令总数 | 38 个 | 收敛至 ~15 个服务接口 |
| 组件直调 invoke 数 | 25 个 (跨 4 组件) | 0 (全部通过 service 层) |
| 服务层 | 不存在 | 5 个领域 service |
| composable 数 | 4 个 (工具类) | 8+ 个 (含业务编排类) |
| 重复代码 | LLM API client ~200 行 ×2, FileEntry 类型 ×3 | 0 |
| Store API 风格 | Composition + Options 混用 | 统一 Composition API |

### 1.2 后端度量

| 指标 | 当前值 | 目标值 |
|------|--------|--------|
| Rust 总代码量 | 15,680 行 / 33 文件 | 保持，拆分密度 |
| Tauri 命令数 | 59 个 | 收敛至 ~20 个 facade |
| manager.rs 行数 | 1,893 | < 400 |
| connection.rs 行数 | 1,643 | < 600 |
| file_ops.rs 行数 | 1,345 (含 18 个 Tauri command) | < 200 (纯 facade) |
| db.rs 行数 | 1,097 | init + migration 分离 |
| 应用服务层 | 不存在 | 4 个领域 service |
| 错误类型 | String 主导 + 4 种局部枚举 | 统一 AppError 分层 |
| DB 迁移策略 | ALTER TABLE 堆叠，无版本追踪 | schema_version + ordered migrations |
| 传输子系统 | 双系统并存 (legacy + new TransferManager) | 统一为 new TransferManager |

### 1.3 架构核心问题

```
当前调用链：
  Component -> invoke('list_files') -> Tauri Command -> SshManager::actor -> SessionSshPool -> ssh2
                    ↑ 无中间层，组件直接了解后端命令名和参数结构

目标调用链：
  Component -> fileService.listPage(id, path) -> invoke('file_list_page', payload) -> FileFacade -> FileService -> SshManager::ops -> SessionSshPool -> ssh2
                    ↑ 组件只了解领域意图         ↑ 前端 service 封装             ↑ Rust facade 转发  ↑ 领域服务编排
```

## 2. 分阶段实施计划

```
Phase 1 (前端 Service 层)

```

// src-tauri/src/application/error.rs
#[derive(Debug, thiserror::Error)]
pub enum AppError {
    #[error("{0}")]
    UserError(String),           // 用户可操作的错误（认证失败、文件不存在等）

    #[error("{0}")]
    Retryable(String),           // 可重试的临时错误（网络超时、连接池耗尽等）

    #[error("{0}")]
    InfraError(String),          // 基础设施错误（系统资源不足、SSH库错误等）

    #[error("不变量违反: {0}")]
    InvariantViolation(String),  // 内部逻辑一致性错误（不应发生）
}

impl AppError {
    pub fn is_retryable(&self) -> bool { matches!(self, Self::Retryable(_)) }
    pub fn user_message(&self) -> &str { ... }
}

impl Into<String> for AppError { ... }  // Tauri command 兼容
```

## Phase 6: 统一错误模型 + 可观测性扩展 (P2)

**目标**：统一前后端错误模型，扩展 transfer observability 到连接生命周期。

**预估工时**：2-3 天
**依赖**：Phase 4 + Phase 5 完成
**风险等级**：低（增量改进）

### Step 6.1: 统一错误分层

**后端**（已在 Phase 3 建立 `AppError`）：
- `UserError` — 展示给用户，带操作建议
- `Retryable` — 自动重试或提示重试
- `InfraError` — 记录日志，展示通用错误
- `InvariantViolation` — 仅日志，触发告警

**前端**（统一错误处理 service）：
- `src/services/errorHandler.ts` — 将后端错误分类，映射到 UI 行为

    pub auth_duration_ms: AtomicU64,         // 认证耗时
    pub reconnect_count: AtomicU64,          // 重连次数
    pub sftp_init_failures: AtomicU64,       // SFTP 初始化失败次数
    pub command_cancel_count: AtomicU64,     // 命令取消次数
    pub pool_exhaustion_count: AtomicU64,    // 连接池耗尽次数
}
```

### Step 6.3: 事件命名规范化

**当前问题**：事件名分散拼接，如 `command-output-{sessionId}-{toolCallId}`

**统一规范**：
```
{domain}:{action}:{detail}

示例：
  connection:status          (替代 connection:status)
  connection:error           (替代 connection:error)
  connection:reconnect       (替代 connection:reconnect)
  transfer:progress          (替代 transfer-progress)
  transfer:error             (替代 transfer-error)
  terminal:data:{sessionId}  (替代 term-data:{sessionId})
  terminal:exit:{sessionId}  (替代 term-exit:{sessionId})
  command:output:{sessionId}:{toolCallId} (替代 command-output-{sessionId}-{toolCallId})
```

**验收标准**：
- [ ] 所有错误路径使用 `AppError` 分层
- [ ] 前端能根据错误类型展示不同 UI
- [ ] 连接生命周期指标可查询
- [ ] 事件名符合 `{domain}:{action}` 规范

### 3.3 风险缓解

| 风险 | 缓解措施 |
|------|---------|
| Phase 1 引入 service 层导致调用路径变长 | service 函数保持薄封装，不增加额外异步层 |
| Phase 2 组件拆分导致 props 传递过深 | 使用 provide/inject 或 composable 共享状态 |
| Phase 3 后端重构导致功能回归 | 保持 Tauri command 签名不变，只改内部实现 |
| Phase 4 manager.rs 拆分引入并发 bug | 保持 actor model 和 channel 通信模式不变，只拆代码组织 |
| Phase 5 DB 迁移失败导致数据丢失 | 每个 migration 在事务中执行，失败自动回滚 |

### 3.4 不做的事

- 不引入新的前端/后端重型依赖
- 不更换框架或语言
- 不改变 Tauri command 的命名和签名（Phase 3 只改内部实现）
- 不改变现有的 actor model 并发模型
- 不重写，只收敛边界
